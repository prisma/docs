[Interactive transactions](https://www.prisma.io/docs/concepts/components/prisma-client/transactions#interactive-transactions)Â allow you to pass an async function into aÂ `$transaction`, and execute any code you like between the individual Prisma Client queries. If your application encounters an error as the transaction is being executed, the function will throw an exception and automatically rollback the transaction.

Here are some of the feature highlights we've built:

- Support for definingÂ [transaction isolation levels](https://www.prisma.io/docs/concepts/components/prisma-client/transactions#transaction-isolation-level)Â â€” fromÂ [`4.2.0`](https://github.com/prisma/prisma/releases/tag/4.2.0)
- Support for the Prisma Data Proxy â€” fromÂ [`4.6.0`](https://github.com/prisma/prisma/releases/tag/4.6.0)

Here's an example of an interactive transaction with aÂ `Serializable`Â isolation level:

You can now remove theÂ `interactiveTransactions`Â Preview feature in your schema.

#### Relation mode

`relationMode="prisma"`Â is now stable for our users working with databases that don't rely on foreign keys to manage relations.Â ğŸ‰

Prismaâ€™sÂ [relation mode](https://www.prisma.io/docs/concepts/components/prisma-schema/relations/relation-mode)Â started as a way to support PlanetScale which does not allow you to create foreign keys for better online migration support. We transformed that into ourÂ *Referential Integrity Emulation*Â inÂ [`3.1.1`](https://github.com/prisma/prisma/releases/tag/3.1.1)Â when we realised that more users could benefit from it, and then integrated it as the default mode for MongoDB, which generally does notÂ *have*Â foreign keys. Prisma needed to use emulation to give the same guarantees.

We then realized the feature was more than just referential integrity and affected how relations work. To reflect this, we renamed the feature to relation mode and theÂ `datasource`Â property toÂ `relationMode`Â inÂ [`4.5.0`](https://github.com/prisma/prisma/releases/tag/4.5.0)

#### Index warnings forÂ `relationMode = "prisma"`

We've added a warning to our Prisma schema validation that informs you that the lack of foreign keys might result in slower performance â€” and that you should add anÂ `@@index`Â manually to your schema to counter that. This ensures your queries are equally fast in relation modeÂ `prisma`Â as they are with foreign keys.

> WithÂ relationMode = "prisma", no foreign keys are used, so relation fields will not benefit from the index usually created by the relational database under the hood. This can lead to slower performance when querying these fields. We recommend manually adding an index.

We also added a fix to ourÂ [VS Code extension](https://marketplace.visualstudio.com/items?itemName=Prisma.prisma)Â to help adding the suggested index with minimal effort:

### General improvements

#### Improved serverless experience with smaller engines size

We have decreased the size of our engine files by an average of 50%. The size of the Query Engine used on Debian, with OpenSSL 3.0.x, for example, went from 39MB to 14MB.
