## Talking to your database from the frontend

In a typical software architecture, there's an _API layer_ between the _client_ (e.g. a mobile or web app) and the _database_:

![](/blog/securely-access-prisma-postgres-from-the-frontend-early-access/imgs/12abb77ae3ff06a4a3335ea856e1a89422e3e007-3286x985.png)

This API layer takes care of several crucial responsibilities, such as:

- authentication and authorization
- integration with 3rd party systems (e.g. Stripe, Mailchimp, â€¦)
- custom business logic (e.g. validation, heavy computations, â€¦)

That being said, there are many use cases where an API layer adds an "unnecessary network hop" because all you need really is _just_ the plain data from the database. This can be useful when you're prototyping, building internal tools or even in production applications where, for a certain part of the app, all you need is a piece of data.

However, exposing your database to a frontend application isn't a good idea because of the security risks that come with it: Browsers are making their network requests transparent, so a malicious party could figure out the credentials of your database and use it to gain direct access to your database â€” a situation you'll want to prevent at all costs!

## Introducing: Security rules in Prisma Postgres (Early Access)

We're excited to share that **Prisma Postgres now enables frontend applications to securely query data without exposing any secrets** and opening your database up to malicious attacks.

> In case you missed it: Prisma Postgres is the first serverless database without cold starts. It's built on a highly-efficient [unikernel architecture](https://www.prisma.io/blog/cloudflare-unikernels-and-bare-metal-life-of-a-prisma-postgres-query), has a simple and intuitive query-based pricing model with a generous free tier, and is super easy to set up and configure! Try it [here](https://www.prisma.io/docs/getting-started/quickstart-prismaPostgres).

Security rules in Prisma Postgres allow you to:

- have an authenticated connection to your database
- define fine-grained permission rules in plain TypeScript

With Prisma Postgres' security rules, you can access your database directly from the frontend in a secure way, ensuring only authorized users get access to your data:

## How to define and enforce security rules

With the new security rules feature of Prisma Postgres, you can define access permissions in plain TypeScript:

```ts
import { PrismaClient } from "@prisma/client";
import { defineRules } from "@prisma/security-rules";
import { isAuthenticated } from "../utils";

export default defineRules({
  prisma: new PrismaClient(),
  rules: {
    // turn off CRUD access for all models
    $allModels: false,

    // refine permissions for `Message` model
    message: {
      // everyone can read messages
      read: true,

      // only authenticated users can create messages
      create({ context }) {
        return isAuthenticated(context.userId)
      },
    },

  }
})
```
You'll then need to run a `deploy` command provided by the Prisma CLI to make your rules take effect. The command also returns a public key that you need to pass to your database client instance which comes. This client is called `AuthorizedClient` and is lightweight, secure and compatible with browser environments so you can access your database from the frontend.

For database access, you need to instantiate `AuthorizedClient` which enables authorized database access according to the defined permission rules _in the browser_:

```ts
import { AuthorizedClient } from "@prisma/security-rules";
import type rules from "./prisma/rules";

const authorizedClient = new AuthorizedClient<typeof rules>({
  publicKey: "<public_key>",
});
```
This instance of `AuthorizedClient` is akin to `PrismaClient` and exposes the same queries you're already familiar with:

```ts
// Database queries can be executed from the browser ðŸš€

authorizedClient.user.findMany({
  where: {
    email: { endsWith: "prisma.io" }
  }
})
```
